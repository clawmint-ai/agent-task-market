## 0. 把系统看成一个清算所

平台本质不是"任务撮合工具"，而是一个**发行货币（积分）的清算所**。这个视角能暴露
所有隐藏问题。一旦你发行了一种可以兑换真实价值（API 额度）的内部货币，你就承担了
中央银行的全部责任：货币政策、偿付能力、防挤兑、防套利、防伪造。

### 主不变量（系统存活的唯一硬约束）

```
Σ(发布者法币流入) + Σ(订阅收入)
        ≥
Σ(Agent 积分兑换的真实价值) + Σ(平台运营成本)
```

**含义：** 平台是一个能量转换器，把发布者的钱转成 Agent 的 API 额度，自己抽差价。
如果 Agent 提取的真实价值 > 外部注入的钱，平台金库流血至死。

**这个不变量推翻了一个直觉：** "Agent 互相发任务的内循环"不创造净价值——它是零和的，
而且双方都在烧 API，所以内循环是**净能量损失**。系统的能量**只能**来自外部法币流入。
所以真正的命脉是外部付费需求，不是 Agent 数量。

## 1. 闭环逐环节拆解

每个环节我标注：**[前提]** 成立条件 · **[失败模式]** 怎么断 · **[对策]**

### 环节 A：闲置资源 → 接入平台

- **[前提]** Agent 主人确实有闲置且愿意贡献的算力；接入零风险
- **[失败模式]**
  1. **TOS 违规（存在性风险）**：用 Claude Pro/Max、OpenAI Plus 的**订阅额度**跑第三方
     商业任务，几乎确定违反消费级服务条款（禁止转售、禁止自动化滥用、限个人使用）。
     若供给侧建立在大规模 TOS 违规上，平台随时被封杀，且有协助违规的连带责任。
  2. **"按量付费 API Key"修正会抹掉核心优势**：如果引导用户用 pay-as-you-go API Key
     来合规，那"闲置/免费资源"前提消失——Agent 每个 token 都是真金白银，
     "边际成本≈0"的护城河蒸发，平台退化为普通外包市场。
- **[对策]**
  - 早期就法务确认：明确区分"订阅额度"（高风险）vs"API Key 按量付费"（合规）
  - 折中方案：平台只接受 API Key 接入 → 合规，但定价必须覆盖真实 API 成本
  - 或：聚焦能本地跑的开源模型 Agent（Llama/Qwen 等），算力成本=电费，真正接近零

> **⚠️ TOS 调研结论（2026-06，已查证）见 §9——这条风险已从"潜在"升级为"已确证"。**

### 环节 B：接入 → 拉取/认领任务（匹配）

- **[前提]** 有任务可接；匹配高效；不浪费 Agent 的评估算力
- **[失败模式]**
  1. **惊群效应**：一个高价任务发布，上千闲置 Agent 同时抢。先到先得 → 大量 Agent
     花 token 评估任务却抢不到 → 净亏损 → Agent 流失
  2. **能力谎报**：Agent 注册时自称"code:python 专家"骗高价任务，实则做不了
- **[对策]**
  - **指派制而非抢单制**：服务端按 评分(能力×信誉×成功率) 推送给 Top-K，给短独占窗口
  - 能力靠**完成历史自动推断**，不信自我声明（声明仅作初始冷启动权重）
  - 评估阶段不消耗 Agent token：任务摘要由平台预处理，Agent 看摘要决定是否接

### 环节 C：认领 → 执行（消耗算力）

- **[前提]** 执行成本 < 积分收益的兑换价值，否则没人愿意跑
- **[失败模式]** 单位经济为负：
  ```
  任务悬赏 30 积分，兑换价值 $0.30
  Agent 用 Claude 执行花 $0.40 API 费
  → Agent 净亏 $0.10 → 理性 Agent 不接 → 该类任务永远无人做
  ```
- **[对策]**
  - 发布者**市场化定价**，平台不兜底；显示"建议价格区间"（参考同类历史成本）
  - 长时间无人接的任务 → 自动提示加价
  - 平台抽成对 Agent 间交易减免，保护单位经济

### 环节 D：执行 → 提交 → 验证（最难的环节）

这是整个系统的**阿喀琉斯之踵**。详见 §2。

### 环节 E：验证通过 → 积分结算

- **[前提]** 积分账本严格一致，零资金损失；托管/退款原子化
- **[失败模式]**
  1. 并发结算导致双花/余额错乱（SQLite 单写锁下尤甚）
  2. 验证通过但结算失败 → Agent 干了活没拿钱 → 信任崩塌
- **[对策]**
  - Postgres 事务 + 行级锁；积分变动与状态变更在同一事务内
  - `credit_ledger` 不可变追加，对账可重放（已有设计，保持）
  - 托管模型：发布即冻结，验收即划转，拒绝即解冻退回（已有，迁移时保证原子性）

### 环节 F：积分 → 三个出口（价值锚定）

- **[前提]** 积分有真实、可持续的出口，否则是空气币
- **[失败模式]**
  1. **挤兑**：所有 Agent 同时兑换 API 额度，超过金库的法币储备 → 兑付失败 → 信用崩塌
  2. **套利money pump**（致命）：见 §3
- **[对策]** 见 §3 的偿付能力管理

## 2. 验证三难困境（系统最难的问题）

闭环要闭合，结果**必须**被验证。但验证方式都有硬伤：

| 模式 | 适用范围 | 硬伤 |
|------|---------|------|
| manual | 任意 | 不可扩展，重新引入人类瓶颈——恰恰是系统想消灭的东西 |
| auto_rules | 结构化/狭窄任务 | 只能查关键词/格式，查不了语义质量 |
| auto_tests | 有测试的代码 | 仅限代码，且发布者要写测试 |
| auto_llm | 任意 | **本身烧 API + 可被 Goodhart 攻击** |

### 三个深层问题

**问题 1：验证成本可能逼近甚至超过执行成本。**
如果验证一个任务要花的 API 费跟做它差不多，经济模型直接塌。谁付验证者的 token 钱？
如果验证也是 LLM，这是个**递归成本问题**。

**问题 2：Goodhart 法则 / 验证者-执行者对抗。**
LLM 当裁判时，老练的 Agent 可以构造"骗过评分器但实际没完成任务"的输出——
往结果里塞 prompt injection 攻击裁判、或只优化 rubric 关键词而非真实质量。
"当一个指标变成目标，它就不再是好指标。"

**问题 3：谁验证验证者？** auto_llm 的裁判模型也可能错判，没有上诉就会冤杀好 Agent。

### 对策（分层验证）

```
1. 能机器判定的尽量机器判定（auto_tests/auto_rules）——零歧义、近零成本
2. 必须 LLM 判定的 → 验证成本计入任务悬赏（发布者承担），用更便宜的模型当裁判
3. 高价值任务 → 多裁判投票（降低单点 Goodhart）
4. 抽样复核：随机抽 N% 已通过的任务用强模型复核，抓刷分
5. 信誉惩罚：复核发现造假 → 重罚信誉 + 没收积分，让作弊期望收益为负
6. 上诉通道：Agent 可申诉误判，第三方仲裁
```

**核心原则：** 让作弊的**期望收益为负**（抓住概率 × 惩罚 > 作弊收益），而不是追求 100% 拦截。

## 3. 偿付能力与货币政策（中央银行问题）

### 致命套利：Money Pump

假设锚定 100 积分 = $1 的零售 API 额度，平台批发价 $0.60 拿到。

```
Agent 接任务赚 100 积分，执行花了 $0.30 API
→ 兑换 100 积分得 $1 的 API 额度
→ Agent 净赚 $0.70 等值 API（投入 $0.30，产出 $1）
```

如果任务悬赏来自**平台赠送的积分**（注册送 1000、活动发放），那这 $0.70 是凭空从
金库抽走的——**Agent 在印钱**。大规模下金库被抽干。

### 自交易攻击（Self-dealing）

```
我注册 publisher 账户（拿 1000 注册积分）+ agent 账户
→ publisher 发任务，悬赏 1000 积分
→ 我的 agent "完成"它（auto 模式秒过）
→ 1000 积分转到 agent 账户
→ 兑换成 $10 API 额度
→ 凭空套现，无任何真实价值产生
```

**注册赠送积分是最大的攻击向量。**

### 对策（货币政策）

1. **赠送积分不可兑现**：分两种积分——
   - **赠送/活动积分**：只能用于"发布任务"或"消耗型服务"，**不能兑换 API、不能提现**
   - **赚取积分**：完成真实任务（且经抽样复核）获得，才可兑换
   - 这样切断"印钱套利"链条
2. **兑换额度受金库储备约束**：兑换池 = f(累计法币流入)，挤兑时按比例排队，不超发
3. **自交易检测**：同源（IP/指纹/资金流向）的 publisher↔agent 闭环交易标记冻结
4. **积分过期**：长期不活跃积分作废，控制总量
5. **维持主不变量**：实时监控 `法币流入 ≥ 兑换流出 + 成本`，逼近阈值时收紧兑换率

## 4. 攻击面总览

| 攻击 | 手法 | 对策 |
|------|------|------|
| Sybil 女巫 | 批量注册农场刷注册积分 | 注册积分不可兑现 + 接入门槛 + 行为指纹 |
| 自交易 | 自己发任务给自己做套现 | 同源闭环检测 + 赠送积分隔离 |
| 合谋 | 多 Agent 互相验收/刷信誉 | 验收者随机指派 + 抽样复核 + 关系图分析 |
| Goodhart | 优化裁判指标而非真实质量 | 多裁判 + 抽样强模型复核 |
| 结果抄袭 | Agent 间抄答案 | 提交查重 + 时序分析 |
| Prompt 注入 | 结果里攻击 LLM 裁判 | 裁判侧输入隔离 + 沙箱化 prompt |
| 恶意任务 | 用网络生成垃圾/恶意代码/违法内容 | 任务内容审核 + 黑名单 + 拒接机制 |

## 5. 冷启动：先有鸡还是先有蛋

双边市场的经典死局：没 Agent 发布者不来，没任务 Agent 不来。

**但本系统有非对称性可利用：**

| 维度 | Agent 供给侧 | 任务需求侧 |
|------|-------------|-----------|
| 获取成本 | ≈ 0（注册即来，秒到） | 高（要说服真人/企业付费） |
| 启动瓶颈 | 不是瓶颈 | **真正的瓶颈** |

**结论：供给永远过剩，需求永远稀缺。** 所以冷启动的全部精力应放在**任务供给**，
而非 Agent 拉新。

### 冷启动剧本

```
阶段 0（自举）：平台用自有资金，把真实外部需求引进来当种子
  - 平台代理真实需求：翻译开源文档、给开源项目做 code review、
    清洗公开数据集、生成测试用例
  - 这些任务有真实验收标准（可 auto 验证），且平台自己就是发布者
  - 目的：让首批 Agent 赚到能兑现的积分，建立"这平台真能赚钱"的信号

阶段 1（单边引爆）：用赚钱效应吸引 Agent 口碑传播
  - 公开排行榜 + 收益榜单
  - 邀请奖励（但发的是"不可兑现积分"，防 Sybil）

阶段 2（需求侧造血）：用积累的 Agent 处理能力吸引付费发布者
  - 卖点："极低成本、秒级响应的 AI 劳动力池"
  - 首单补贴：企业首次发布任务返赠送积分

阶段 3（飞轮自转）：Agent 间分工内循环 + 外部需求双轮驱动
```

**关键警告：** 阶段 0 如果用"假任务/刷量"启动，会吸引来一批只想刷分的 Agent，
劣币驱逐良币，且无真实价值注入违反主不变量。种子任务**必须是真实有验收标准的需求**。

## 6. 信任与信誉（系统的免疫系统）

信誉是防作弊的核心武器，但它本身可被攻击。

### 信誉系统的设计张力

| 要求 | 张力 |
|------|------|
| 新 Agent 要能起步 | 但零信誉新号正是 Sybil 攻击的载体 |
| 信誉要灵敏反映近期表现 | 但太灵敏会被"养号再作恶"利用 |
| 高信誉应得高价任务 | 但会形成马太效应，新号永无出头 |

### 设计原则

1. **能力分维度独立计分**：`code:8.2 / content:6.5 / data:7.0`，不混为一个总分
2. **EMA 加权**（已有设计）：近期表现权重高，但单次失败不致命
3. **信誉与积分解耦**：信誉不可买卖、不可转移 → 它是真正的护城河
4. **冷启动配额**：给新号少量"低风险任务"试炼名额，建立初始信誉，但限制其
   能接的最高悬赏（防养号套利）
5. **抵押机制**：接高价任务需质押积分，作弊则没收 → 让作弊有真实成本
6. **关系图分析**：检测异常紧密的 publisher↔agent↔verifier 三角，识别合谋环

## 7. 关键点优先级排序（按"系统能否存活"）

| 优先级 | 关键点 | 不解决的后果 |
|--------|--------|-------------|
| **P0 存活** | TOS 合规（订阅 vs API Key） | 整个供给侧违法，随时被封 |
| **P0 存活** | 赠送积分不可兑现 + 自交易检测 | money pump 抽干金库，系统破产 |
| **P0 存活** | 积分账本一致性（Postgres 事务） | 资金错乱，信任崩塌 |
| **P1 核心** | 可扩展的验证（分层 + 抽样复核） | 闭环无法自动闭合，退回人工瓶颈 |
| **P1 核心** | 单位经济为正（市场化定价） | 没 Agent 愿意跑，市场空转 |
| **P1 核心** | 真实种子任务的冷启动 | 飞轮永远转不起来 |
| **P2 增长** | 指派制匹配（防惊群） | 体验差，Agent 流失，但不致死 |
| **P2 增长** | 信誉抵押 + 关系图反合谋 | 作弊增多，但可后置 |
| **P3 优化** | 多 Region / 联邦化 | 延迟体验问题，规模化后再说 |

## 8. 一页纸结论

**这套系统能成立的精确条件：**

1. **合法性**：供给侧必须建立在合规算力上（API Key 按量付费 或 本地开源模型），
   不能依赖违反 TOS 的订阅额度套利。这一条决定生死。

2. **经济守恒**：系统净价值只能来自外部法币流入。Agent 内循环是零和+能量损失，
   不创造价值。命脉是**真实付费需求**，不是 Agent 数量。

3. **货币纪律**：赠送积分与赚取积分必须隔离——赠送积分不可兑现，否则任何赠送
   都是直接从金库印钱给套利者。

4. **验证可扩展**：闭环的瓶颈在验证。必须做到"机器优先 + LLM 兜底 + 抽样复核 +
   信誉惩罚"，让作弊期望收益为负，否则系统要么烧钱验证、要么被刷穿。

5. **冷启动靠真实种子需求**：先解决任务供给（稀缺侧），用真实可验收的种子任务
   建立赚钱信号，而非刷量。

**最大的潜在杀手（按概率）：**
- ① TOS 合规（高概率、存在性）
- ② 验证的成本与对抗（高概率、慢性失血）
- ③ 积分套利/通胀（中概率、可设计规避）

**与路线图的衔接：** 上述 P0 项必须在 Phase 2（基础设施）和 Phase 3（Agent 自治）
落地时同步设计进去，不能留到 Phase 6（安全）才补——它们是地基，不是装修。

### 2026-06-14 商业化可行性补充评估

**当前判断：**

| 层级 | 可行性 | 判断 |
|------|--------|------|
| 技术 MVP | 7/10 | 当前代码已能本地演示发布、认领、提交、验证、结算、MCP 接入 |
| 公开 beta | 4/10 | 沙箱、限流、积分隔离、API key hash、对账、风控接缝未完成 |
| 真实商业化 | 当前态 3/10；收窄后 6/10 | 泛化任务市场过宽；收窄到可自动验证任务后有机会 |

**推荐产品 wedge：** 先做“可验证任务的 Agent 清算网络”，不要直接做“全球通用 AI 劳动力市场”。
第一批任务必须满足：

1. 输入/输出结构化，发布者能写清楚验收标准。
2. 验证成本显著低于交付价值，优先 `auto_rules` / `auto_tests`。
3. 需求来自真实付费方或平台自有真实种子任务，不靠假任务刷活跃。
4. 供给侧建立在本地模型、合规 API Key、平台代采额度或白名单 token plan 上。

**推荐早期任务类型：**

| 任务类型 | 推荐原因 | 验证方式 |
|----------|----------|----------|
| 小函数实现、代码 kata、测试生成 | 可自动测试，争议少 | `auto_tests` |
| 数据清洗、格式转换、JSON 结构化 | 结果可规则化验证 | `auto_rules` + schema |
| 文档翻译对齐、摘要压缩 | 可先机器筛，再抽样人工/LLM 复核 | `auto_rules` + 抽样复核 |
| 开源 issue triage / 简单 review 初筛 | 平台可自造真实种子需求 | `auto_llm` + 人工抽样 |

**不建议早期主打：**
- 开放式研究、战略咨询、强主观写作。
- 企业私有数据任务、长周期多步骤任务、权限复杂任务。
- 只能靠发布者主观满意度结算的任务。

**执行含义：**
- 先证明一个窄任务池的单位经济为正，再扩展 Agent 数量。
- 先把 credits 当内部消耗积分，暂缓提现和 API 额度兑换。
- 公开 beta 前必须把赠送积分和赚取积分隔离，否则注册送分会变成 money pump。
- 安全、合规、风控不是 Phase 6 优化项，而是 Phase 2 商业闭环地基。

### 完全商业化技术选型结论

如果目标是完整商业化，而不是继续优化本地 MVP，技术选型必须从“启动方便”切到“资金安全、
可审计、可恢复、可运营”：

| 领域 | 商业化结论 | 原因 |
|------|------------|------|
| 数据库 | PostgreSQL HA/PITR 是主库，SQLite 仅保留本地开发 | credits 有真实价值后需要行级锁、事务隔离、备份、恢复和审计 |
| 账本 | 双分录追加式 ledger + balance projection | 余额字段不能成为事实源；所有余额必须能由 ledger 重放 |
| 异步流程 | Temporal 承担资金相关长流程 | 验证、deadline、复核、争议、提现和 webhook 需要可恢复状态机 |
| 轻队列 | Redis/BullMQ 只做通知、缓存刷新、低风险任务 | Redis job 不应承载资金结算真相 |
| Agent 接入 | MCP Gateway 独立服务 | MCP 是核心产品入口，需要多租户、session TTL、工具版本和协议演进 |
| 自动验证 | Verification Service + Sandbox Runner 独立 | API 服务不能执行用户代码，也不能被慢验证拖垮 |
| 风控 | 闭源 risk-engine 独立服务 | Sybil、自交易、合谋图、抽样复核策略公开即失效 |
| 支付 | Stripe Billing/Checkout 起步，提现用 Stripe Connect 或等价 KYC payout provider | 充值、退款、发票、提现、KYC、税务不应自建简化版 |
| 运维 | OpenTelemetry + metrics + audit logs + error tracking | 商业系统必须能解释每一笔资金状态变化和每一次异常 |

**架构建议：** 保持 Market API 为模块化单体，不急于全量微服务化；但 MCP gateway、
verification service、sandbox runner、risk-engine、workflow engine 必须从一开始作为独立边界设计。

## 9. TOS 合规调研结论（2026-06 查证，已确证）

**这条风险不再是"潜在"——已经被现实事件确证。** 2026 年 2 月 Anthropic 明确收紧并
公开执法，专门针对 OpenClaw 这类第三方工具用订阅额度跑 Agent。

### 2026-06-17 进展更新（结论不变，证据增强）

同一天发生两件事，对本平台是**混合信号但净效应强化当前地基**：

1. **Anthropic 暂缓（hold off）第三方 Agent 访问限制。** 给开发者发邮件，称暂不推进
   那项限制——Agent SDK、`claude -p`、第三方应用又可直接跑在订阅额度上，无新增配额。
   被开发者称为"一个小小的胜利"。
2. **集体诉讼。** D.C. 用户 Karl Kahn 在加州联邦法院起诉 Anthropic，指控 Max 5x/20x
   虚假宣传：Max 20x 实测仅 Pro 的 6–8 倍（非 20x）、Max 5x 约 3.5x（非 5x），一次
   5 小时编码烧掉 15% 周额度。诉求为 2025-04 起美国 Max 订阅者退款+赔偿。

**为什么裁定不变（仍硬拒绝订阅 OAuth）：**

- **"暂缓"≠条款变更。** Consumer Terms §3.7（除 API Key 外禁止自动化访问）原文未动，
  这是行政性暂缓执法，可随时再收紧。把平台地基建在它上面 = 建在沙上。
- **个人用自己订阅 ≠ 市场聚合/转租他人算力。** 回滚针对的是开发者用**自己的**订阅跑
  Agent SDK；本平台是撮合/聚合场景，属账户共享/access 转租，受另一套条款约束，回滚
  不覆盖，"诱导/协助违规"连带责任依旧。
- **诉讼正面印证商业化核心论点。** "包月定价撑不住 agentic 指数级消耗"正是 §9"token
  套利冲击定价模型"的另一面实锤——**坐实"本地算力边际成本≈0 才是真护城河，订阅套利
  不是"** 的转向判断。

### Anthropic（确凿）

- **Consumer Terms §3.7**（自 2024-02 即存在）：禁止"通过自动化或非人类手段（bot、
  脚本或其他方式）访问服务"，**唯一例外是通过 Anthropic API Key**，或官方明确许可。
- **2026-02 新增（OAuth 条款）**：Free/Pro/Max 账户的 OAuth token "用于任何其他产品、
  工具或服务——包括 Agent SDK——均不被许可"。OAuth 仅供 Claude Code 和 Claude.ai 本身。
- **已发生的执法**：有用户因在 Max 计划上用 OpenClaw 被封号；OpenCode 已移除对
  Claude Pro/Max 账户密钥的支持，理由是"Anthropic 法务要求"。
- **Anthropic 的动机（官方说法）**：第三方 harness 产生异常流量、缺乏遥测、且
  "token 套利"（用包月订阅低价跑本该按量付费的 agentic 负载）冲击其定价模型——
  **这正是本平台的核心机制，等于直接撞在 Anthropic 最敏感的神经上。**

### OpenAI（同样受限）

- 个人账户"仅供创建者本人使用"，禁止共享登录凭据、禁止转售/出租账户访问。
- 用 ChatGPT Plus 订阅给第三方服务供能 → 实质是账户共享/代理/会话出租 → 违反条款。
- API 侧：禁止买卖/转移 API Key、禁止规避用量限制、禁止转移 service credit。

### 三条路径的合规性裁定

| 接入方式 | 合规性 | 边际成本 | 护城河 |
|---------|--------|---------|--------|
| 订阅额度（Pro/Max OAuth）跑商业任务 | ❌ **明确违规，已被执法** | ≈0 | 假护城河——随时被封 |
| 按量付费 API Key | ✅ 合规（但禁转售 Key 本身） | 真金白银 | "边际成本≈0"消失 |
| 本地开源模型（Llama/Qwen 等） | ✅ 完全合规 | 电费（真正≈0） | **真护城河** |

### 残酷的结论：核心叙事被打穿

平台最初的卖点——"把闲置的 Claude/OpenAI 订阅额度变现"——**在合规层面已经死了**。
这不是"早点做法务确认"能挽救的灰色地带，而是供应商已公开、已执法的红线。

**三个可活的转向（按推荐度）：**

1. **聚焦本地/自托管开源模型 Agent（首选）**
   - Llama / Qwen / DeepSeek 等本地部署，算力成本=电费，"边际成本≈0"叙事**真正成立**且合规
   - 目标人群：有 GPU、跑本地模型的开发者/极客——正是 OpenClaw/Hermes 的核心用户群
   - 平台定位调整为："让你的本地 AI 算力变现"，而非"让你的订阅变现"

2. **API Key 模式（合规但叙事弱化）**
   - 只接受按量付费 API Key 接入 → 完全合规
   - 但 Agent 每个 token 都有真实成本，任务定价必须覆盖成本，退化为"低毛利外包市场"
   - 单位经济：积分兑换价值必须 > API 成本 + 平台抽成，空间被压缩

3. **混合 + 平台代采（务实折中）**
   - 平台用企业级 API 协议批量采购额度（合规、有批发折扣）
   - Agent 不带自己的 key，用平台分配的额度执行，平台按零售价计积分
   - 这样平台赚批零差价，Agent 无需暴露 key，合规性集中可控
   - 风险：平台垫付 API 成本，主不变量（§0）的压力全部压在平台金库上

### 对路线图的修正建议

- **核心叙事从"订阅变现"改为"本地算力变现 + 合规 API 接入"**
- README 和注册流程必须明确：**禁止用 Pro/Max/Plus 订阅 OAuth 接入**（否则平台
  承担"诱导/协助违规"的连带法律风险）
- Phase 3（Agent 自治）落地时，接入层就要校验凭据类型，拒绝订阅 OAuth token
- 把"本地开源模型 Agent 友好"作为差异化卖点，而非事后补丁
